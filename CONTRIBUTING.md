## Development Guidelines

### Code Style

- Follow the existing patterns in the repository
- Use meaningful variable and function names
- Comment complex logic, not obvious code
- Keep functions focused and small

### Commit Messages

We follow conventional commits:

- `feat:` â€“ new feature
- `fix:` â€“ bug fix
- `docs:` â€“ documentation only
- `style:` â€“ code style changes (formatting, missing semicolons, etc.)
- `refactor:` â€“ code change that neither fixes a bug nor adds a feature
- `test:` â€“ adding missing tests
- `chore:` â€“ changes to build process, dependencies, etc.

Example: `feat: add Stripe webhook signature verification`

### Testing

- Write tests for new functionality
- Ensure existing tests pass
- Test edge cases where relevant

## ğŸ”„ Pull Request Process

1. **Fork the repository** (for external contributors)
2. **Create a feature branch** from `main`
3. **Make your changes**, following the guidelines above
4. **Write or update tests** as needed
5. **Update documentation** if you're changing behavior
6. **Submit a pull request** with a clear description of what you've done
7. **Respond to feedback** â€“ we may request changes

For our private templates, we primarily accept 'Feature Requests' via GitHub Discussions. If you have a fix for a private repo, please share the code snippet in a private support email to support@keelstack.me so we can review and merge it into the core product for everyone.

### PR Requirements

- All automated checks must pass
- At least one maintainer review required
- No merge conflicts with `main`
