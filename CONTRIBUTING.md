## 🚀 First Time? Start Here
New to open source or contributing on GitHub? We're glad you're here!
1.  **Explore**: Look for issues labeled `good first issue` or `help wanted` in our [public community repo](https://github.com/keelstack-me/community).
2.  **Ask**: Feel free to ask questions in the relevant issue or discussion thread—no question is too small.
3.  **Use AI Tools**: You may use AI assistants (e.g., GitHub Copilot, Cursor) to draft code or documentation. However, you must follow our [AI Contributions Policy](#-ai-contributions-policy).

Contributors are responsible for reviewing and validating any AI-generated code or content before submission.

## 💡 How Can I Contribute?

### 🤖 AI Contributions Policy
We embrace AI as a tool, but we require a **human in the loop** for all contributions.
- **Disclosure**: You must disclose any substantial use of AI in your pull request description or commit message, specifying which tool(s) you used.
- **Accountability**: You are fully responsible for any code or content you submit, regardless of whether it was generated by AI. You must understand and be able to explain the code.
- **Quality**: We will reject pull requests that we deem low‑quality or "AI slop". Do not waste maintainers' time with code you haven't personally reviewed and tested.

### 🐛 Reporting Bugs
If you find a bug in any KeelStack template:
1.  **Check Existing Issues:** Someone may have already reported it. Quick search saves everyone time.
2.  **Use the Bug Report Template:** It helps us diagnose faster. Please include:
    *   Your environment (e.g., Next.js version, Node version, OS).
    *   Clear steps to reproduce the issue.
    *   What you expected to happen vs. what actually happened.
    *   Relevant error logs or screenshots.
    *   If possible, indicate severity (blocking, major, minor) and whether the issue affects production usage.

### ✨ Suggesting Features
Have an idea for a new "AI‑Native" feature?
1.  Check the **Ideas** category in our [GitHub Discussions](https://github.com/keelstack-me/community/discussions/categories/ideas) first—someone else might have had the same thought.
2.  If it's new, create a new discussion. Describe:
    *   **The problem:** What's a pain point for solo founders that this solves?
    *   **The solution:** How do you envision the feature working?
    *   **Why it fits:** How does it align with KeelStack's AI‑native, modular philosophy?

### 📚 Improving Documentation
Good documentation is as important as good code. Found a typo? An unclear explanation? A missing example?
*   **Small fixes:** You can suggest a change directly via a pull request on the relevant repository.
*   **Larger improvements:** Open an issue or discussion first to get feedback.

## 🛠️ Development Guidelines
Following these guidelines helps us review and merge your contributions faster.

### 🎨 Code Style & AI‑Readiness
*   **Follow Existing Patterns:** Consistency is key. Look at the surrounding code and match its style.
*   **Write for Humans and AI:** Use clear, descriptive variable and function names. Keep functions focused and small. This makes your code easier for both people **and AI assistants (like Cursor/Claude)** to understand and work with.
*   **Comment the "Why," Not the "What":** Explain *why* a piece of logic exists, especially if it's non‑obvious. The "what" should be clear from the code itself.

### 💬 Commit Messages
We use the [Conventional Commits](https://www.conventionalcommits.org) standard. This helps automate changelog generation.
*   `feat:` – A new feature (e.g., `feat: add payment provider webhook handler`).
*   `fix:` – A bug fix (e.g., `fix: handle webhook signature timeout`).
*   `docs:` – Documentation only changes.
*   `refactor:` – A code change that neither fixes a bug nor adds a feature.
*   `test:` – Adding missing tests or correcting existing tests.
*   `chore:` – Changes to the build process or auxiliary tools.

**Example:**
```text
feat(auth): add OAuth support for Google login
Implement OAuth flow for Google authentication, including session management and user profile syncing.
Closes #42
```

### 📝 Code Review Process
All contributions will be reviewed by a maintainer.
- **Be responsive**: Expect feedback within 3–5 business days. Please respond to comments and questions promptly.
- **Keep PRs small**: Focused, small pull requests are easier and faster to review.
- **Be open to feedback**: Code review is a collaborative process. Not all suggestions will be accepted, but all will be considered.

## 🔄 Pull Request (PR) Process

1.  **Fork the Repository** (for public community tools).
2.  **Create a Feature Branch:** Branch from `main` and give it a descriptive name (e.g., `fix/stripe-webhook-error`, `feat/add-lemon-squeezy`).
3.  **Make Your Changes:** Follow the development guidelines above.
4.  **Test Your Changes:** Ensure your changes don't break existing functionality.
5.  **Update Documentation:** If you're changing behavior, update the relevant documentation files or README.
6.  **Submit the Pull Request:**
    *   Fill out the PR template clearly.
    *   Link any related issues (e.g., "Closes #12").
    *   Explain the "why" behind your change – what problem does it solve, and why is this the best approach?
7.  **Review & Iterate:** A maintainer will review your PR. Be open to feedback and make adjustments as needed. This is a collaborative process!

## 🛡️ For Pro Template Contributors

Our core SaaS starter kits are **private, paid products**. We deeply appreciate contributions that make these kits better for everyone. Here's how you can help:

1.  **Feature Requests & Ideas:** Please use the [Ideas category in Discussions](https://github.com/keelstack-me/community/discussions/categories/ideas). This is the best place to propose and refine new features.
2.  **Reporting Bugs in Pro Kits:** If you find a bug in a kit you've purchased, please email us directly at **[support@keelstack.me](mailto:support@keelstack.me)** with details. This ensures your report is handled quickly and securely.
3.  **Submitting Fixes:** If you've developed a fix for a pro kit, we'd love to see it! Please email the code snippet or a patch file to **[support@keelstack.me](mailto:support@keelstack.me)**.
4.  **Rewards:**
    *   High‑quality, accepted bug fixes for our core products are rewarded with **free lifetime updates** for the affected kit.
    *   Significant feature contributions may be rewarded with **special discounts** or even a **free copy of another KeelStack kit** of your choice.
